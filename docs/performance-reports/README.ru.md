## Сводка отчёта

В отчёте зафиксированы результаты измерений CPU, памяти, потоков, структуры
процесса, запуска, сетевых запросов и стабильности процесса Crypto HUD в
указанных ниже условиях.

При стандартной конфигурации с тремя рыночными парами и интервалом обновления
пять секунд в ходе десятиминутного запуска с прогретым кэшем средняя загрузка
CPU относительно всей машины составила **0,070%**, медиана частной
зафиксированной памяти — **20,27 MiB**, медиана рабочего набора —
**47,96 MiB**, медиана частного рабочего набора — **19,29 MiB**.

При 20 рыночных парах в реальном времени и том же интервале средняя загрузка
CPU составила **0,125%**, CPU P95 — **0,681%**, медиана частной
зафиксированной памяти — **21,56 MiB**, медиана рабочего набора —
**49,02 MiB**.

Ни в одном измеренном сценарии не создавались дочерние процессы или процессы
WebView2. Память процесса после скрытия в системный трей была близка к
показателям видимого окна в автономном сценарии. В десятиминутном запуске
память увеличивалась во время прогрева и обновления пятиминутных свечей, после
чего оставалась в узком диапазоне на 7–10-й минутах.

Значения относятся к процессу Crypto HUD на указанной тестовой машине. В них
не входят использование GPU, стоимость композиции DWM и потребляемая
электрическая мощность.

## Определения метрик

- **CPU всей машины:** процессорное время процесса, разделённое на прошедшее
  время и 12 логических процессоров. Соответствует способу отображения общего
  процента в Диспетчере задач Windows.
- **CPU P95:** 95% образцов CPU в установившемся режиме не превышают это
  значение.
- **Частная зафиксированная память:** зафиксированная память, назначенная только
  этому процессу.
- **Рабочий набор:** страницы, находящиеся в RAM; может включать общие страницы.
- **Частный рабочий набор:** страницы в RAM, принадлежащие только процессу.
- **Число потоков:** число потоков процесса в момент каждого образца.
- **Число дескрипторов:** число дескрипторов Windows, принадлежащих процессу.

CPU вычислялся по разнице `TotalProcessorTime`. Из обычных сценариев
установившегося режима исключались первые 10 секунд, из пятиминутного запуска —
первые 30 секунд, из десятиминутного — первая минута.

После обнаружения целочисленного округления в исходной сводке значения CPU в
автономных сценариях были восстановлены из счётчика в эквиваленте ядра и
приведены как округлённые приблизительные значения. Значения сетевых сценариев
в реальном времени не затронуты этой проблемой.

## Наблюдавшиеся различия между сценариями

- При открытом окне настроек медиана частной зафиксированной памяти изменилась
  с 20,27 MiB до 23,74 MiB, а рабочего набора — с 47,96 MiB до 55,35 MiB.
  Длительность сценариев различалась, поэтому это сравнение наблюдавшихся
  значений.
- При увеличении числа живых пар с 3 до 20 средняя загрузка CPU изменилась с
  0,070% до 0,125%, частная зафиксированная память — с 20,27 MiB до
  21,56 MiB, рабочий набор — с 47,96 MiB до 49,02 MiB.
- В автономном режиме 10 окон Quote Board использовали на 2,70 MiB больше
  частной зафиксированной памяти и на 3,55 MiB больше рабочего набора, чем один
  видимый виджет.
- По одному экземпляру пяти видов виджетов использовали на 9,20 MiB больше
  частной зафиксированной памяти и на 11,44 MiB больше рабочего набора, чем
  один автономный виджет.
- Масштаб 300% дал на 1,28 MiB больше частной зафиксированной памяти и на
  2,37 MiB больше рабочего набора по сравнению со стандартным масштабом.
- При скрытии в трей частная зафиксированная память изменилась с 17,43 MiB до
  17,43 MiB, рабочий набор — с 41,93 MiB до 41,77 MiB. Округлённые оценки CPU
  составили 0,04% для видимого и 0,03% для скрытого состояния.

## Измерение запуска

В сценарии запуска с живой сетью приложение достигло существующего маркера
готовности всех рыночных данных за **1,705 секунды**.

Для этого маркера данные должны присутствовать во всех настроенных рыночных
строках. Время включает обнаружение плагинов, создание окон, начальные
рыночные запросы и секундный таймер UI, принимающий рыночные события. Это не
время первого кадра или появления первого окна.

При проверке обновлений в рабочем режиме запрос GitHub Releases API завершился
ошибкой в тестовой среде, поэтому был пройден путь обработки ошибки. Процесс
продолжил работу; успешный ответ обновления не измерялся.

## Наблюдения за сетью и файлами

Приблизительное число запросов в установившемся режиме:

`число уникальных пар × (60 / интервал обновления в секундах + 0,2)`
запроса в минуту.

Член `0,2` соответствует одному обновлению свечей каждые пять минут.

| Конфигурация | Приблизительно запросов в минуту |
| --- | ---: |
| 3 пары, обновление 5 секунд | 36,6 |
| 8 пар, обновление 5 секунд | 97,6 |
| 20 пар, обновление 5 секунд | 244 |
| 3 пары, обновление 60 секунд | 3,6 |
| 20 пар, обновление 60 секунд | 24 |

При запуске добавляется примерно один запрос цены и один запрос свечей на
каждую пару. Сетевая задержка удлиняет цикл, поэтому наблюдаемая частота может
быть ниже расчётной.

Локальный прокси, отклонявший соединения, дал четыре неудачных рыночных цикла
за 90 секунд. Сценарий охватывал немедленный отказ соединения, но не
восьмисекундный тайм-аут чтения и не восстановление после сбоя.

После запуска и прогрева измеренная активность чтения и записи файлов в
состоянии простоя была фактически нулевой. Задержка перетаскивания и сохранения
на медленном диске не измерялась. Счётчики I/O процесса Windows не
использовались как счётчики сетевых байтов процесса.

- Наблюдавшиеся процессы WebView2: 0.

## Объём и ограничения теста

- Тестовая машина использовала QEMU и удалённый виртуальный видеостек.
  Физический GPU, композиция DWM и питание от батареи не измерялись.
- Каждый специальный сценарий запускался один раз. Доверительные интервалы по
  нескольким запускам отсутствуют.
- Максимальная длительность составляла 10 минут. Запуск на 8–24 часа не
  выполнялся.
- Не измерялись системный DPI 150–300%, конфигурация 4K с несколькими
  мониторами, отключение дисплея, сон и возобновление, непрерывная анимация с
  минимальным интервалом.
- Сетевой сбой создавался немедленным отказом прокси. Не измерялись задержка
  DNS, медленные ответы, HTTP 429, восьмисекундный тайм-аут и восстановление
  сети.
- Атрибуция через Windows Performance Recorder и WPA была недоступна. Нет
  данных о сетевых байтах процесса, GPU, переключениях контекста, отдельных
  пробуждениях и электрической мощности.
- Не измерялась задержка сохранения состояния при перетаскивании и изменении
  настроек на медленном или синхронизируемом диске.
- Результаты относятся к версии 0.9.7 и хешу исполняемого файла в начале
  отчёта.

## Запись о проверке

- Сборка release:

- Проверка подготовленного процесса выпуска со встроенными плагинами и
  ресурсами.
- Семь детерминированных автономных сценариев.
- Семь сценариев с живой сетью.
- Один десятиминутный запуск с прогретым кэшем.
- Измерения CPU, трёх определений памяти, потоков, дескрипторов, файлового I/O,
  готовности запуска, дочерних процессов и пути ошибки.
- Проверка фактов документа, навигации, относительных ссылок, UTF-8 и изоляции
  двунаправленного текста в арабской версии.

Поведение продукта во время выполнения не изменялось в процессе сбора данных.
