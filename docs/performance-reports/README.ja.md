## レポート概要

このレポートは、下記のテスト条件で測定した Crypto HUD プロセスの
CPU、メモリ、スレッド、プロセス構成、起動時間、ネットワーク要求、
安定性に関するデータを記録したものです。

市場ペア 3 件、更新間隔 5 秒の既定構成をウォームキャッシュ状態で
10 分間実行した結果、マシン全体に対する平均 CPU は **0.070%**、
プライベートコミットの中央値は **20.27 MiB**、ワーキングセットの
中央値は **47.96 MiB**、プライベートワーキングセットの中央値は
**19.29 MiB** でした。

同じ 5 秒間隔でライブ市場ペアを 20 件にした測定では、平均 CPU
**0.125%**、CPU P95 **0.681%**、プライベートコミット中央値
**21.56 MiB**、ワーキングセット中央値 **49.02 MiB** を記録しました。

測定した全シナリオで、子プロセスおよび WebView2 プロセスは生成
されませんでした。トレイに格納した状態のプロセスメモリは、
オフラインでウィンドウを表示した状態とほぼ同じ値でした。10 分間の
実行では、ウォームアップと 5 分足更新の時間帯にメモリが増加し、
7～10 分目には狭い範囲に収まりました。

数値は、記載したテストマシン上の Crypto HUD プロセスに対するもの
です。GPU 使用率、DWM 合成処理、消費電力は含まれていません。

## 指標の定義

- **マシン全体の CPU:** プロセス CPU 時間を経過時間と 12 個の論理
  プロセッサで除した値です。Windows タスク マネージャーのマシン
  全体の百分率と同じ方式です。
- **CPU P95:** 定常状態の CPU サンプルの 95% がこの値以下です。
- **プライベートコミット:** 当該プロセスだけに割り当てられた
  コミット済みメモリです。
- **ワーキングセット:** 現在 RAM に常駐するページです。共有ページを
  含む場合があります。
- **プライベートワーキングセット:** RAM に常駐し、当該プロセスだけが
  使用するページです。
- **スレッド数:** 各サンプル時点のプロセススレッド数です。
- **ハンドル数:** プロセスが所有する Windows ハンドル数です。

CPU は `TotalProcessorTime` の差分から算出しました。通常の定常状態
シナリオでは最初の 10 秒、5 分実行では最初の 30 秒、10 分実行では
最初の 1 分をサンプルから除外しました。

初回の生集計で整数丸めの問題が見つかったため、オフライン CPU 値は
コア換算カウンターから再構成した丸め済みの概算値です。ライブ
ネットワークの CPU 値には、この丸めの問題はありません。

## シナリオ間で観測された差

- 設定ウィンドウ表示時、プライベートコミット中央値は 20.27 MiB から
  23.74 MiB、ワーキングセット中央値は 47.96 MiB から 55.35 MiB に
  変化しました。シナリオの実行時間が異なるため、これは観測値の
  比較です。
- ライブペア数を 3 から 20 にすると、平均 CPU は 0.070% から
  0.125%、プライベートコミット中央値は 20.27 MiB から 21.56 MiB、
  ワーキングセット中央値は 47.96 MiB から 49.02 MiB に変化しました。
- オフラインの Quote Board 10 ウィンドウは、表示ウィジェット 1 個
  よりプライベートコミットが 2.70 MiB、ワーキングセットが 3.55 MiB
  多く測定されました。
- 5 種類のウィジェットを各 1 個表示した場合、単一ウィジェットより
  プライベートコミットが 9.20 MiB、ワーキングセットが 11.44 MiB
  多く測定されました。
- 300% スケールでは、既定スケールのオフラインシナリオより
  プライベートコミットが 1.28 MiB、ワーキングセットが 2.37 MiB
  多く測定されました。
- トレイ格納により、プライベートコミットは 17.43 MiB から
  17.43 MiB、ワーキングセットは 41.93 MiB から 41.77 MiB に変化
  しました。丸め済みオフライン CPU 概算値は、表示時 0.04%、
  格納時 0.03% でした。

## 起動時間

ライブ起動シナリオで、アプリケーションは既存の全市場データ準備完了
マーカーに **1.705 秒**で到達しました。

このマーカーは、設定された市場行すべてにデータがあることを条件と
します。時間にはプラグイン検出、ウィンドウ生成、初回市場要求、
市場イベントを受け取る 1 秒間隔の UI タイマーが含まれます。
最初のフレームや最初のウィンドウ表示時間ではありません。

本番相当の更新確認では、テスト環境で GitHub Releases API 要求が
失敗したためエラー経路を通りました。プロセスは実行を継続し、
成功した更新応答は測定していません。

アプリケーションは処理を継続し、終了コード 0 で終了しました。

## ネットワークとファイルの観測

定常状態の 1 分当たり要求数の概算式は次のとおりです。

`固有ペア数 × (60 / 更新間隔（秒） + 0.2)` 要求/分

`0.2` は 5 分ごとのローソク足更新 1 回を表します。

| 構成 | 1 分当たりの概算要求数 |
| --- | ---: |
| 3 ペア、5 秒更新 | 36.6 |
| 8 ペア、5 秒更新 | 97.6 |
| 20 ペア、5 秒更新 | 244 |
| 3 ペア、60 秒更新 | 3.6 |
| 20 ペア、60 秒更新 | 24 |

起動時には、各ペアにつき概ね価格要求 1 回とローソク足要求 1 回が
加わります。ネットワーク遅延でサイクルが延びるため、観測要求数は
式の値を下回る場合があります。

接続を拒否するローカルプロキシでは、90 秒間に 4 回の市場サイクル
失敗を記録しました。このシナリオは即時接続拒否を対象とし、8 秒の
読み取りタイムアウトや障害後の再接続は測定していません。

起動とウォームアップ後、アイドルシナリオのファイル読み書き活動は
実質的にゼロでした。低速ディスクでのドラッグ操作および設定保存の
遅延は測定していません。Windows のプロセス I/O カウンターは、
プロセスに帰属するネットワークバイトとして使用していません。

## プロセス構成

- 観測された子プロセス: 0。
- 観測された WebView2 プロセス: 0。
- 既定ライブ時のスレッド数: 中央値 9、最大値 13。
- ライブ 20 ペア時のスレッド数: 中央値 14、最大値 18。
- 既定ライブ時のハンドル数: 中央値 301、最大値 322。

## テスト範囲と制約

- テストマシンは QEMU とリモート仮想ディスプレイスタックを使用
  しました。物理 GPU 使用率、DWM 合成コスト、バッテリー電力は
  測定していません。
- カスタムシナリオは各 1 回実行しました。複数回実行による信頼区間
  は含まれていません。
- 最長実行時間は 10 分です。8～24 時間の実行は行っていません。
- システム DPI 150～300%、4K マルチモニター、ディスプレイ切断、
  スリープ復帰、最短間隔での連続アニメーションは測定していません。
- ネットワーク障害はプロキシの即時拒否を使用しました。DNS 停滞、
  低速応答、HTTP 429、8 秒タイムアウト、ネットワーク復旧は測定
  していません。
- Windows Performance Recorder と WPA による帰属分析は利用できません
  でした。プロセス別ネットワークバイト、GPU 使用率、コンテキスト
  スイッチ回数、独立ウェイクアップ回数、消費電力は含まれていません。
- 低速または同期対象ディスクでのドラッグ中および設定変更時の状態
  保存遅延は測定していません。
- 結果はバージョン 0.9.7 とレポート冒頭の実行ファイルハッシュに
  対応します。
