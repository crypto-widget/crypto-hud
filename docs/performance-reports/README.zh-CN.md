## 报告摘要

本报告记录 Crypto HUD 在下述测试条件中的 CPU、内存、线程、进程结构、启动、
网络请求和稳定性数据。

默认配置为 3 个交易对、每 5 秒刷新一次。热缓存运行 10 分钟时，记录到的平均
整机 CPU 为 **0.070%**，私有提交内存中位数为 **20.27 MiB**，工作集中位数为
**47.96 MiB**，私有工作集中位数为 **19.29 MiB**。

在 20 个真实交易对、相同 5 秒刷新间隔下，平均 CPU 为 **0.125%**，CPU P95
为 **0.681%**，私有提交内存中位数为 **21.56 MiB**，工作集中位数为
**49.02 MiB**。

所有受测场景都没有创建子进程或 WebView2 进程。托盘隐藏后的进程内存与离线可见
场景接近。10 分钟测试中的内存在预热和 5 分钟 K 线刷新阶段上升，并在第 7–10
分钟保持在一个稳定区间。

本报告中的数值描述指定测试机上的 Crypto HUD 进程，不包含归因到 GPU、DWM 合成
或实际电功耗的数据。

## 指标说明

- **整机 CPU：**进程 CPU 时间除以实际时间和 12 个逻辑处理器，对应 Windows
  任务管理器常用的整机百分比口径。
- **CPU P95：**稳态样本中 95% 的 CPU 数值不高于该值。
- **私有提交：**只分配给该进程的已提交内存。
- **工作集：**当前驻留在 RAM 中的内存页，可能包含共享页。
- **私有工作集：**该进程独占的驻留 RAM，最接近任务管理器常见的进程内存数值。
- **线程数：**每个采样点的进程线程总数；临时网络线程会使该值变化。
- **句柄数：**进程持有的 Windows 句柄数量。

CPU 使用 `TotalProcessorTime` 增量计算。普通稳态场景排除前 10 秒；5 分钟场景
排除前 30 秒；10 分钟场景排除第一分钟。

首次离线汇总存在整数舍入问题，因此离线 CPU 值根据单核等效计数器重新换算为近似值。
真实联网 CPU 值不受该舍入问题影响。

## 场景结果

CPU 数值为整台 12 逻辑处理器测试机的百分比。

| 场景 | CPU 平均 / P95 | 私有提交中位数 | 工作集中位数 | 线程中位数 / 最大值 |
| --- | ---: | ---: | ---: | ---: |
| 默认真实联网，3 个交易对，热缓存 10 分钟 | 0.070% / 0.189% | 20.27 MiB | 47.96 MiB | 9 / 13 |
| 设置窗口打开并真实联网 | 0.071% / 0.233% | 23.74 MiB | 55.35 MiB | 14 / 17 |
| 20 个真实交易对，5 秒刷新 | 0.125% / 0.681% | 21.56 MiB | 49.02 MiB | 14 / 18 |
| 本地代理立即拒绝连接 | 0.049% / 0.232% | 18.66 MiB | 43.67 MiB | 14 / 17 |
| 1 个可见组件，确定性离线行情 | 约 0.04% / 0.25% | 17.43 MiB | 41.93 MiB | 12 / 12 |
| 同一组件通过托盘隐藏并停放到屏幕外 | 约 0.03% / 0.25% | 17.43 MiB | 41.77 MiB | 12 / 12 |
| 10 个 Quote Board 窗口，离线 | 约 0.14% / 0.50% | 20.13 MiB | 45.48 MiB | 12 / 12 |
| 五种组件各运行一个 | 约 0.17% / 0.50% | 26.63 MiB | 53.37 MiB | 12 / 12 |
| 1 个 Quote Board 显示 20 个离线交易对 | 约 0.09% / 0.42% | 17.87 MiB | 42.78 MiB | 12 / 12 |
| 单个组件缩放至 300%，离线 | 约 0.08% / 0.50% | 18.71 MiB | 44.30 MiB | 12 / 12 |

最高一次应用 CPU 单样本出现在 20 个真实交易对场景，为 **0.916%**。

## 场景间观测差异

- 设置窗口场景的私有提交内存中位数为 23.74 MiB，10 分钟默认场景为
  20.27 MiB；工作集中位数分别为 55.35 MiB 和 47.96 MiB。两个场景持续时间
  不同，因此这里只陈述观测值。
- 真实交易对从 3 个增加到 20 个后，平均 CPU 从 0.070% 变为 0.125%，私有提交
  从 20.27 MiB 变为 21.56 MiB，工作集从 47.96 MiB 变为 49.02 MiB。
- 离线场景中，10 个 Quote Board 比 1 个可见组件多使用 2.70 MiB 私有提交和
  3.55 MiB 工作集。
- 五种组件各运行一个时，比离线单组件场景多使用 9.20 MiB 私有提交和
  11.44 MiB 工作集。
- 300% 缩放比离线默认缩放多使用 1.28 MiB 私有提交和 2.37 MiB 工作集。
- 托盘隐藏前后私有提交均为 17.43 MiB；工作集从 41.93 MiB 变为 41.77 MiB。
  舍入后的离线 CPU 估值分别为 0.04% 和 0.03%。

## 启动测量

真实联网启动场景在 **1.705 秒**后达到现有的“全部行情数据就绪”标记。

该标记要求已配置的行情行都有数据。时间包含插件发现、窗口创建、首批行情请求，以及
每秒一次、用于消费行情事件的 UI 定时器。该数值不是首帧或首次显示窗口时间。

生产式更新检查在测试环境中访问 GitHub Releases API 失败，因此该场景记录的是失败
路径。进程继续运行；成功更新响应没有测量。

## 网络与文件观测

稳态请求量近似为：

`唯一交易对数量 × (60 / 刷新秒数 + 0.2)` 次/分钟。

`0.2` 表示每 5 分钟一次 K 线刷新。

| 配置 | 估算请求数/分钟 |
| --- | ---: |
| 3 个交易对，5 秒刷新 | 36.6 |
| 8 个交易对，5 秒刷新 | 97.6 |
| 20 个交易对，5 秒刷新 | 244 |
| 3 个交易对，60 秒刷新 | 3.6 |
| 20 个交易对，60 秒刷新 | 24 |

启动时，每个交易对大约增加一次 ticker 请求和一次 K 线请求。网络延迟会延长单个
周期，因此实际请求速率可能低于公式结果。

本地代理拒绝连接场景在 90 秒内记录到 4 个失败行情周期。该场景覆盖立即拒绝连接，
没有覆盖 8 秒读取超时或断网后重连。

启动和预热后，静置场景记录到的文件读写基本为零。没有测量慢磁盘上的拖动保存和设置
保存延迟。Windows 进程 I/O 计数器没有用作进程网络字节统计。

## 测试范围与限制

- 测试机使用 QEMU 和远程虚拟显示，未量化物理 GPU、DWM 合成和电功耗。
- 每个自定义场景运行一次，本报告不包含多轮置信区间。
- 最长运行时间为 10 分钟，没有执行 8–24 小时测试。
- 未测量系统 150–300% DPI、4K 多显示器、显示器移除、睡眠唤醒和最短间隔持续动画。
- 网络故障场景使用代理立即拒绝连接；未测量 DNS 卡顿、慢响应、HTTP 429、8 秒超时
  和网络恢复。
- 当前环境无法使用 Windows Performance Recorder 和 WPA 归因。本报告不包含进程
  网络字节、GPU 利用率、上下文切换、独立唤醒次数和电功耗。
- 未在慢速或同步磁盘上测量拖动与设置保存延迟。
- 结果只对应报告顶部所列的 0.9.7 版本和可执行文件哈希。
