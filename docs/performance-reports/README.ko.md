## 보고서 요약

이 보고서는 아래 테스트 조건에서 측정한 Crypto HUD 프로세스의 CPU,
메모리, 스레드, 프로세스 구조, 시작 시간, 네트워크 요청 및 안정성
데이터를 기록합니다.

기본 구성인 시세 페어 3개와 5초 새로 고침 간격으로 10분간 웜 캐시
상태를 측정한 결과, 전체 시스템 기준 평균 CPU는 **0.070%**, 전용
커밋 메모리 중앙값은 **20.27 MiB**, 작업 집합 중앙값은 **47.96 MiB**,
전용 작업 집합 중앙값은 **19.29 MiB**였습니다.

같은 5초 간격에서 실시간 시세 페어를 20개로 설정한 측정에서는 평균
CPU **0.125%**, CPU P95 **0.681%**, 전용 커밋 메모리 중앙값
**21.56 MiB**, 작업 집합 중앙값 **49.02 MiB**가 기록되었습니다.

측정한 모든 시나리오에서 자식 프로세스와 WebView2 프로세스는
생성되지 않았습니다. 트레이로 숨긴 상태의 프로세스 메모리는 오프라인
가시 창 시나리오와 유사했습니다. 10분 측정에서는 워밍업과 5분 봉
갱신 구간에 메모리가 증가한 뒤 7~10분에 좁은 범위를 유지했습니다.

수치는 명시된 테스트 컴퓨터에서 실행된 Crypto HUD 프로세스에
해당합니다. GPU 사용량, DWM 합성 비용, 전력은 측정 범위에 포함되지
않았습니다.

## 측정 지표 정의

- **전체 시스템 CPU:** 프로세스 CPU 시간을 경과 시간과 논리 프로세서
  12개로 나눈 값입니다. Windows 작업 관리자의 전체 시스템 비율
  방식과 같습니다.
- **CPU P95:** 정상 상태 CPU 표본의 95%가 이 값 이하임을 뜻합니다.
- **전용 커밋:** 해당 프로세스에만 할당된 커밋 메모리입니다.
- **작업 집합:** 현재 RAM에 상주하는 메모리 페이지이며 공유 페이지가
  포함될 수 있습니다.
- **전용 작업 집합:** 현재 RAM에 상주하며 해당 프로세스에만 속한
  페이지입니다.
- **스레드 수:** 각 표본 시점의 프로세스 스레드 수입니다.
- **핸들 수:** 프로세스가 소유한 Windows 핸들 수입니다.

CPU는 `TotalProcessorTime` 차이로 계산했습니다. 일반 정상 상태
시나리오의 처음 10초, 5분 실행의 처음 30초, 10분 실행의 처음 1분은
표본에서 제외했습니다.

초기 원시 요약에서 정수 반올림 문제가 확인되어 오프라인 CPU 값은
코어 등가 카운터로 재구성한 반올림 근삿값입니다. 실시간 네트워크
CPU 값에는 이 반올림 문제가 없습니다.

## 시나리오 간 관측 차이

- 설정 창을 열었을 때 전용 커밋 중앙값은 20.27 MiB에서 23.74 MiB로,
  작업 집합 중앙값은 47.96 MiB에서 55.35 MiB로 변했습니다. 시나리오
  실행 시간이 서로 다르므로 두 값은 관측된 측정치입니다.
- 실시간 페어 수가 3개에서 20개로 늘 때 평균 CPU는 0.070%에서
  0.125%로, 전용 커밋 중앙값은 20.27 MiB에서 21.56 MiB로, 작업 집합
  중앙값은 47.96 MiB에서 49.02 MiB로 변했습니다.
- 오프라인 Quote Board 창 10개는 가시 위젯 1개보다 전용 커밋
  2.70 MiB, 작업 집합 3.55 MiB가 더 높았습니다.
- 위젯 5종을 각각 1개씩 실행한 시나리오는 단일 오프라인 위젯보다
  전용 커밋 9.20 MiB, 작업 집합 11.44 MiB가 더 높았습니다.
- 300% 배율 위젯은 기본 배율 오프라인 위젯보다 전용 커밋 1.28 MiB,
  작업 집합 2.37 MiB가 더 높았습니다.
- 트레이로 숨겼을 때 전용 커밋은 17.43 MiB에서 17.43 MiB로, 작업
  집합은 41.93 MiB에서 41.77 MiB로 변했습니다. 반올림된 오프라인 CPU
  근삿값은 표시 상태 0.04%, 숨김 상태 0.03%였습니다.

## 시작 측정

실시간 시작 시나리오에서 애플리케이션은 기존의 전체 시세 데이터 준비
표시에 **1.705초** 만에 도달했습니다.

이 표시는 구성된 각 시세 행에 데이터가 있어야 성립합니다. 측정 시간은
플러그인 검색, 창 생성, 최초 시세 요청, 시세 이벤트를 소비하는 1초
간격 UI 타이머를 포함합니다. 첫 프레임이나 첫 창 표시 시간은 아닙니다.

운영 방식의 업데이트 확인은 테스트 환경에서 GitHub Releases API
요청이 실패하여 오류 경로를 따랐습니다. 프로세스는 계속 실행되었고,
성공한 업데이트 응답은 측정하지 않았습니다.

## 네트워크 및 파일 관측

정상 상태의 분당 요청 수 근삿값은 다음과 같습니다.

`고유 페어 수 × (60 / 새로 고침 초 + 0.2)`회/분

`0.2`는 5분마다 수행되는 봉 갱신 1회를 나타냅니다.

| 구성 | 분당 요청 수 근삿값 |
| --- | ---: |
| 3개 페어, 5초 새로 고침 | 36.6 |
| 8개 페어, 5초 새로 고침 | 97.6 |
| 20개 페어, 5초 새로 고침 | 244 |
| 3개 페어, 60초 새로 고침 | 3.6 |
| 20개 페어, 60초 새로 고침 | 24 |

시작 시 페어마다 시세 요청 약 1회와 봉 요청 약 1회가 추가됩니다.
네트워크 지연은 한 주기를 늘릴 수 있으므로 관측 요청률은 계산식보다
낮을 수 있습니다.

연결을 거부한 로컬 프록시에서는 90초 동안 실패한 시세 주기가 4회
기록되었습니다. 이 시나리오는 즉시 연결 거부만 다뤘으며 8초 읽기
시간 초과와 네트워크 복구 후 재연결은 측정하지 않았습니다.

시작과 워밍업 이후 유휴 시나리오에서 측정한 파일 읽기 및 쓰기는
사실상 0이었습니다. 느린 디스크에서 드래그 및 설정 저장 지연은
측정하지 않았습니다. Windows 프로세스 I/O 카운터는 프로세스별
네트워크 바이트로 사용하지 않았습니다.

## 테스트 범위 및 제한 사항

- 테스트 컴퓨터는 QEMU와 원격 가상 디스플레이 스택을 사용했습니다.
  물리 GPU 사용량, DWM 합성 비용, 배터리 전력은 측정하지 않았습니다.
- 사용자 지정 시나리오는 각각 한 번 실행했습니다. 반복 실행 신뢰
  구간은 포함하지 않습니다.
- 최장 실행 시간은 10분이며 8~24시간 실행은 수행하지 않았습니다.
- 시스템 DPI 150~300%, 4K 다중 모니터, 모니터 제거, 절전/복귀,
  최소 간격 연속 애니메이션은 측정하지 않았습니다.
- 네트워크 장애는 즉시 프록시 거부로 만들었습니다. DNS 지연, 느린
  응답, HTTP 429, 8초 시간 초과, 네트워크 복구는 측정하지 않았습니다.
- Windows Performance Recorder 및 WPA를 이용한 귀속 측정은 사용할 수
  없었습니다. 프로세스별 네트워크 바이트, GPU 사용량, 컨텍스트 전환
  수, 독립적인 깨우기 수, 전력은 포함하지 않습니다.
- 느리거나 동기화되는 디스크에서 드래그 및 설정 변경 중 상태 저장
  지연을 측정하지 않았습니다.
- 결과는 버전 0.9.7과 보고서 상단의 실행 파일 해시에 해당합니다.
